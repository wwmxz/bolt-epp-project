## 一、推荐的最终方案（Final）：LBR 基线 + 目标化渐进插桩收集 Transition Sidecar + BOLT Path-aware Reorder

### 目标

在真实应用程序上以低开销获得稳定的 **edge heat（fdata）**，同时以可控预算渐进式收集 **edge→edge transitions**（路径相关性），并让 BOLT 重排算法显式使用 transitions，把热路径片段更紧凑地布局，从而提升 I-cache/BTB/前端效率。

---

## 二、整体架构（3 条数据流 + 2 个产物）

### 产物 A：`baseline.fdata`（只来自 LBR/perf）

- 来源：perf branch stack/LBR → perf2bolt → fdata
- 语义：可靠边热度基线（**不掺入 EPP/插桩的派生边权**）
- 用途：BOLT 传统重排/其它优化 pass 的主 profile 输入

### 产物 B：`transition.sidecar`（来自目标化/渐进插桩）

- 来源：LLVM 插桩（受 LBR 引导，按轮次、按预算）→ merge → sidecar
- 语义：edge→edge 的 transition 计数 + 置信度/覆盖信息
- 用途：BOLT 重排算法的附加输入（仅影响布局决策，不改变 baseline.fdata）

### 核心思想：Profile 职责分离

- **edge heat** = 由硬件采样来保证低开销与真实性
- **edge→edge transitions** = 由插桩专门收集“相关性/连续性”，并按预算渐进覆盖
- BOLT 重排算法 = 同时消费两者（edge heat 仍是主信号，transition 为补充约束/奖励）

## 三、关键模块与接口设计（你需要实现/修改的东西）

### Module 1：LBR 基线采集（已有工作流）

1) perf record（带 branch stack）
2) perf2bolt 生成 `baseline.fdata`
3) BOLT 生成 `bolt_opt_baseline`（作为对照）

> 这部分尽量用现成链路，保持“工业可跑”。

---

### Module 2：Hot Edge Planner（每轮生成插桩计划）

**输入**

- `baseline.fdata`
- 可选：BOLT CFG（从 BOLT 输出或内部拿到 edge 列表/BB 列表）
- 轮次参数：`round_id, budget, policy`

**输出**

- `plan_round_k.json`
  - 需要包含：要插桩的 edge 集合（对应 branch 指令集合）、采集强度（采样率/是否启用）、轮次元数据

**策略建议（可写成论文点）**

- Round 1（hot-first）：选 top-K 热边 + 邻域（例如每条热边扩 1-hop/2-hop）  
- Round 2+（coverage/uncertainty-driven）：
  - 添加：高热边附近但 transition 信息缺失的边
  - 添加：冷区中“结构上关键”的边（如 dominator 边、loop backedge、函数入口附近）
- 冷边“更多轮插桩”的实现方式：不是每轮全量插桩，而是**轮换覆盖**（coverage scheduling）

**预算模型（必须明确）**

- 建议至少提供两种 budget：
  - `max_edges_to_instrument`
  - `max_overhead_target`（通过在线采样估计或试运行校准）

### Module 3：LLVM 目标化插桩（收集 edge→edge transition）

你希望“把 EPP 融合到 BOLT 项目中，基于 LLVM 工具插桩”，我建议 V1 做成一个 LLVM pass（更易迭代与测试），后续再决定是否迁入 BOLT 源树。

**3.1 关键要求：统一的 Edge ID 映射（这是成败关键）**
你要能让以下三者对齐同一套 edge ID：

- BOLT/perf 的 binary edge（地址级）
- LLVM 插桩时看到的 CFG edge（IR 级）
- sidecar 中写的 edge ID（BOLT 能回查）

**推荐的 V1 映射方式（工程可控、可渐进完善）**

- 在 LLVM codegen/asm 输出阶段，为每个 machine basic block（或每个分支点）发射一个**可识别的符号/注释**，并生成一个 map 文件：
  - `func_name, bb_label(or offset), succ_bb_label(or offset) -> edge_id`
- BOLT 侧用符号/地址解析反推出 edge_id  
- 插桩 pass 也能根据同样的 edge_id 插到对应 branch

这一步你可以先在“可控 benchmark/测试套件”验证，再扩展到真实应用（符号/调试信息缺失时要降级策略）。

**3.2 插桩采集的数据结构（低开销优先）**
你最终目标是 edge→edge transition，最轻量的做法是：

- 对每个线程维护 `last_edge_id`（TLS）
- 每执行一个被插桩 edge（或 branch），做：
  - `transition_count[last_edge_id, curr_edge_id]++`
  - `last_edge_id = curr_edge_id`
- 计数结构建议：
  - 线程本地 hash map（key=64-bit packed pair），结束时合并
  - 或固定大小的稀疏表（只对 plan 中的 pair 开桶，降低 hash 开销）

**3.3 事件定义**
你说“path 信息表达为 transition(edge→edge)”，那就必须规定：transition 的“边”是什么边？

- 建议定义为：**taken branch edge**（只记录实际走到的 succ）
- 对间接跳转/ret/call：
  - V1 可以先忽略或只处理条件分支
  - V2 扩展支持（更复杂但价值高）

### Module 4：Sidecar 格式与合并（multi-round）

#### 4.1 sidecar 推荐格式（稳定 + 可扩展）

建议用一行一条 transition：

- `src_edge_id,dst_edge_id,count,round_id,coverage,confidence,hotness_context`

其中：

- `count`：本轮观测到的 transition 次数
- `confidence`：可由采集时长、采样率、碰撞率、是否跨线程合并等计算
- `coverage`：该 transition 在多少轮出现/出现频率（合并时再算更合适）

为了后续 BOLT 读取效率，建议同时提供一个二进制格式（例如简单的 varint 编码），但 V1 可以先 CSV/TSV，先把 pipeline 跑通。

#### 4.2 合并策略（替代当前仓库“重复 verify 合并 sidecar”那种）

你想要的是“渐进式覆盖”，合并策略要能表达：

- **总计数**：跨轮 sum（注意：如果每轮采集窗口不同/采样率不同，需要归一化）
- **置信度加权**：count 可以不变，但给 BOLT 的权重 = f(confidence)
- **覆盖奖励**：新出现的 transition 给额外关注（在 BOLT 评分里体现，而不是篡改 baseline fdata）

### Module 5：修改 BOLT 重排算法消费 transitions（你方案核心）

你要做的是：**在 BOLT 的布局决策中，引入 transition 奖励项**。

#### 5.1 你需要在 BOLT 里新增/扩展的数据结构

- `EdgeHeat`：来自 baseline.fdata（已有）
- `TransitionProfile`：从 sidecar 加载，提供：
  - `getTransitionWeight(edge1, edge2) -> w`
  - `getOutgoingTransitions(edge1) -> list(edge2, w)`

#### 5.2 最小可行的算法改动（V1 推荐）

以 ext-tsp 或其链构造/合并过程为例（概念上）：

- 原先主要使用 `w_edge(u->v)` 来评价将块 v 接到块 u 后的收益
- 现在加入：
  - `Score(u->v | prev_edge) = w_edge(u->v) + λ * w_trans(prev_edge -> (u->v))`

其中：

- `prev_edge` 可以用“当前链最后一步的边”近似（不追踪完整历史，避免状态爆炸）
- `λ` 可配置，且可以随 confidence 调节：
  - `λ_eff = λ * clamp(confidence, 0..1)`

这个设计优点：

- 不需要完整 path 状态机（保持工程复杂度可控）
- 直接把“热路径片段连续性”注入布局决策
- 退化容易（λ=0 回到原算法）

#### 5.3 你需要避免的坑

- 不要让 transition 信号盖过 edge heat（否则会布局极端化）
- 要处理“transition 缺失”的情况（默认 0）
- 线程并发/跨核噪声导致的 transition 偏差：用 confidence 限制影响

## 四、渐进式采集策略（真正区别于当前仓库 multiround 的点）

你要的多轮应当是：**每一轮插桩计划不同**，而不是重复同一个 verify。

推荐的 Round 设计（示例）：

- **Round 0**：仅 perf(LBR) → baseline.fdata（无插桩）
- **Round 1（Hot-core）**：插桩 top 1% 热边所在函数 + 1-hop 邻域边，收集 transitions
- **Round 2（Hot-expand）**：插桩 top 5% 热边函数，补齐 Round1 未覆盖的关键分支
- **Round 3（Cold-coverage）**：按“未覆盖边/未覆盖 loop/backedge/未覆盖 dominator edges”轮换插桩
- **Round 4+（Stopping）**：根据 improvement/coverage 收敛停止

每轮输出 sidecar，合并为 `transition.sidecar.merged`，再跑 BOLT path-aware reorder。

## 五、评测与实验设计（确保你的研究说得清）

1) Baseline：perf(LBR) fdata + 原始 BOLT 重排
2) 对照 1：perf(LBR) fdata + 现有“只看边”的算法（同 1）
3) 实验组：perf(LBR) fdata + transitions sidecar + 修改后的重排
4) ablation：
   - 不同 λ
   - 不同预算（每轮插桩边数）
   - 不同 rounds 策略（hot-first vs coverage-first）
   - transitions 来自热区 vs 冷区

指标：

- wall time / throughput
- 前端相关硬件计数（icache miss、branch miss）如果能拿到更好
- sidecar 覆盖率：覆盖了多少 edge、多少 edge-pair
- 插桩开销（slowdown）

## 六、里程碑（建议的落地顺序）

**M1：定义 ID + sidecar 格式 + BOLT 读取 sidecar（不改算法）**  
先把数据能从插桩产出并被 BOLT 读到。

**M2：实现最小 reorder 改动（λ transition bonus）+ 可开关**  
让 sidecar 真正影响布局，能看到差异。

**M3：实现 Planner（热边引导）+ 多轮不同 plan**  
这一步才是你想要的“渐进式”。

**M4：完善映射与鲁棒性（真实应用、缺符号时降级）**
